# 5.1 What is Terminology Binding?

Section [4.2.3 Data Entry Context](/pages/createpage.action?spaceKey=DOCCTXSIG&title=4.2.3+Data+Entry+Context&linkCreation=true&fromPageId=278301526), noted that the environment or use case in which data is entered may:

  1. Determine the range of [concepts](https://confluence.ihtsdotools.org/display/DOCGLOSS/concept "Glossary link: concepts") that can rationally be entered
     * For example, limiting the range of concepts that can be used for diagnoses to subtypes of [64572001 |Disease (disorder)|](http://snomed.info/id/64572001).
  2. Significantly affect the interpretation of _concepts_ and other data entered
     * For example, entry of a subtype of [64572001 |Disease (disorder)|](http://snomed.info/id/64572001) in a "Family History" section should be interpreted as a family history of that disorder, rather than a diagnosis suffered by the subject of the record.

[Terminology bindings](https://confluence.ihtsdotools.org/display/DOCGLOSS/Terminology+binding "Glossary link: Terminology bindings") are links between terminology components and information model artifacts, such as EHR structures and data entry contexts. There are two types of  _terminology bindings_ , [value set bindings](https://confluence.ihtsdotools.org/display/DOCGLOSS/value+set+binding "Glossary link: value set bindings") and [meaning bindings](https://confluence.ihtsdotools.org/display/DOCGLOSS/meaning+binding "Glossary link: meaning bindings").
